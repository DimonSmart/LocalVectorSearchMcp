# 0004.spec-index-storage-and-search

## Intent

The index stores Markdown documents, elements, chunks, lexical rows, vector rows, and manifest metadata in SQLite and serves lexical, semantic, and hybrid search.

## Behavior

Each server instance stores its single project's documents, Markdown elements, chunks, FTS5 index rows, sqlite-vec vectors, and index manifest metadata in one SQLite database. Documents and chunks are identified by project-relative path and contain no knowledge-base column. The manifest records schema version `2`, embedding model, effective embedding dimensions, chunker version, embedding text builder version, chunking maximum bytes, chunking maximum elements, whether heading context is included in embedding text, and whether frontmatter is included in chunks.

Deleting a document removes its document, element, chunk, FTS, and vector rows as one consistent operation. Every operation that accesses `chunk_vectors`, including schema creation or removal, vector writes, vector deletion, and vector search, first initializes sqlite-vec on that connection. Initialization failures are reported as understandable vector-index errors rather than raw native or SQLite exceptions.

Schema initialization and status inspection must not overwrite an existing manifest. Before reindexing, the product compares every required manifest value with the current configuration and implementation versions. A missing or different required value makes an existing index incompatible.

Reindex scope `changed` scans Markdown files selected by the configured include and exclude glob patterns, computes normalized content hashes, skips unchanged files, indexes new or changed files, and deletes files that are no longer selected or no longer exist. Changing include or exclude patterns is an ordinary source-set change and does not require `Force`. Missing or different chunking manifest values make an existing index incompatible. Reindex scope `all` reindexes all selected Markdown files and deletes missing or newly excluded files. `Force` permits complete index recreation when the manifest is incompatible; it does not recreate a compatible index. Without `Force`, reindexing an incompatible index fails with an understandable error that identifies the incompatible values and recommends a forced rebuild. A forced rebuild recreates dimension-dependent vector storage, writes the current manifest, and indexes all discovered documents using the current chunking configuration.

Reindex and search always operate on the one configured project source; there is no all-knowledge-bases mode or knowledge-base filter.

If embedding generation fails while reindexing a changed document, previously indexed content for that document remains intact.

Every embedding vector must match the effective configured dimensions. A mismatch fails with an understandable embedding-provider error before the vector is written.

Lexical search uses SQLite FTS5 and BM25 ranking. Vector search uses sqlite-vec for nearest-neighbor search. Production semantic ranking must not use handwritten vector scans or handwritten cosine similarity. Hybrid search combines semantic and lexical ranks with Reciprocal Rank Fusion, deduplicating chunks present in both lists.

Indexed Markdown slice reads start inclusively at an existing semantic pointer. They return whole elements in ordinal order, with a hard element-count limit and a best-effort UTF-8 byte limit that includes separators between elements. The first element is always returned even when it alone exceeds the byte limit. If an element remains unread because of either limit, `NextPointer` identifies that first unread element so a subsequent read continues without gaps or duplicates. A missing path or starting pointer is reported as pointer-not-found; limits alone must not produce that error.

## Architecture And Patterns

Infrastructure owns SQLite connection creation, migrations, repository operations, FTS5 operations, sqlite-vec integration, embedding provider calls, file scanning, and the indexing pipeline.

Core owns search DTOs, ranking logic, exceptions, and interfaces:

- `IEmbeddingProvider`
- `IVectorIndexService`
- `IFullTextSearchService`
- `IKnowledgeRepository`
- `IKnowledgeBaseIndexer`
- `IKnowledgeSearchService`
- `ISemanticPointerReader`
- `IMarkdownElementParser`
- `IMarkdownChunker`

Embeddings are part of the chunk index. The MVP must not create a separate `embedding_cache` table.

## Non-goals

Manual vector scanning in production, separate embedding cache tables, direct score normalization between BM25 and vector distance, and embedding reuse across files by hash are excluded.

## Acceptance Criteria

- SQLite contains documents, elements, chunks, FTS rows, vector rows, and manifest metadata.
- SQLite schema version is `2`; `documents` and `chunks` contain no `knowledge_base` column and document paths are unique.
- `chunks_fts.rowid` equals `chunks.id`.
- `chunk_vectors.rowid` equals `chunks.id`.
- FTS5 queries are parameterized and escaped.
- Semantic search uses sqlite-vec.
- Hybrid search uses RRF with ranks starting at 1.
- Reindex skips unchanged files, reindexes changed files, removes deleted files, and preserves old content when embedding generation fails.
- Changed reindex removes documents that stop matching include/exclude glob patterns without requiring a forced rebuild.
- Document deletion leaves no related rows in `elements`, `chunks`, `chunks_fts`, or `chunk_vectors`.
- Every `chunk_vectors` operation uses a connection initialized with sqlite-vec; initialization failure raises an understandable vector-index error.
- Manifest metadata includes schema version, chunker version, embedding text builder version, embedding model, effective embedding dimensions, chunking maximum bytes, chunking maximum elements, include-heading-context, and include-frontmatter values.
- Missing or different chunking manifest values make the index incompatible and require `Force` for reindexing.
- Initialization and status inspection do not hide manifest incompatibility by rewriting existing values.
- Reindexing an incompatible index without `Force` fails with the mismatched manifest values and forced-rebuild guidance.
- Reindexing an incompatible index with `Force` recreates all index tables, including dimension-dependent vector storage, writes the current manifest, and indexes all discovered documents.
- `Force` does not recreate a compatible index.
- Embedding vectors whose dimensions differ from the effective configured dimensions are rejected before storage.
- A schema-1 index is incompatible without `Force`; forced reindex recreates it as the current project's schema-2 index.
- Slice reads are inclusive, return no more than the requested element count, do not split elements to meet the byte limit, and return the first unread element as `NextPointer`.
- An oversized first element remains readable, while nonexistent paths and pointers produce a pointer-not-found error.

## Verification

- Integration-test migrations, FTS5, vector table operations, indexing (including removal after include/exclude changes and physical deletion), deletion row consistency, search services, and semantic pointer reader.
- Integration-test model and dimension incompatibility, forced recreation, current manifest persistence, and initialization without manifest mutation.
- Unit-test RRF ranking.
- Integration-test element- and byte-limited slice pagination, continuation from `NextPointer`, oversized first elements, and missing paths or pointers.
