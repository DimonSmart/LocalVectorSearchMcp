# 0004.spec-index-storage-and-search

## Intent

The index stores Markdown documents, elements, chunks, lexical rows, vector rows, and manifest metadata in SQLite and serves lexical, semantic, and hybrid search.

## Behavior

SQLite stores documents, Markdown elements, chunks, FTS5 index rows, sqlite-vec vectors, and index manifest metadata. The manifest records the schema version, embedding model, effective embedding dimensions, chunker version, and embedding text builder version.

Schema initialization and status inspection must not overwrite an existing manifest. Before reindexing, the product compares every required manifest value with the current configuration and implementation versions. A missing or different required value makes an existing index incompatible.

Reindex scope `changed` scans configured Markdown files, computes normalized content hashes, skips unchanged files, indexes new or changed files, and deletes missing files from the index. Reindex scope `all` reindexes all existing Markdown files and deletes missing files. `Force` permits complete index recreation when the manifest is incompatible; it does not recreate a compatible index. Without `Force`, reindexing an incompatible index fails with an understandable error that identifies the incompatible values and recommends a forced rebuild. A forced rebuild recreates dimension-dependent vector storage and writes the current manifest before indexing all discovered documents.

If embedding generation fails while reindexing a changed document, previously indexed content for that document remains intact.

Every embedding vector must match the effective configured dimensions. A mismatch fails with an understandable embedding-provider error before the vector is written.

Lexical search uses SQLite FTS5 and BM25 ranking. Vector search uses sqlite-vec for nearest-neighbor search. Production semantic ranking must not use handwritten vector scans or handwritten cosine similarity. Hybrid search combines semantic and lexical ranks with Reciprocal Rank Fusion, deduplicating chunks present in both lists.

## Architecture And Patterns

Infrastructure owns SQLite connection creation, migrations, repository operations, FTS5 operations, sqlite-vec integration, embedding provider calls, file scanning, and the indexing pipeline.

Core owns search DTOs, ranking logic, exceptions, and interfaces:

- `IEmbeddingProvider`
- `IVectorIndexService`
- `IFullTextSearchService`
- `IKnowledgeRepository`
- `IKnowledgeBaseIndexer`
- `IKnowledgeSearchService`
- `ISemanticPointerReader`
- `IMarkdownElementParser`
- `IMarkdownChunker`

Embeddings are part of the chunk index. The MVP must not create a separate `embedding_cache` table.

## Non-goals

Manual vector scanning in production, separate embedding cache tables, direct score normalization between BM25 and vector distance, and embedding reuse across files by hash are excluded.

## Acceptance Criteria

- SQLite contains documents, elements, chunks, FTS rows, vector rows, and manifest metadata.
- `chunks_fts.rowid` equals `chunks.id`.
- `chunk_vectors.rowid` equals `chunks.id`.
- FTS5 queries are parameterized and escaped.
- Semantic search uses sqlite-vec.
- Hybrid search uses RRF with ranks starting at 1.
- Reindex skips unchanged files, reindexes changed files, removes deleted files, and preserves old content when embedding generation fails.
- Manifest metadata includes schema version, chunker version, embedding text builder version, embedding model, and effective embedding dimensions.
- Initialization and status inspection do not hide manifest incompatibility by rewriting existing values.
- Reindexing an incompatible index without `Force` fails with the mismatched manifest values and forced-rebuild guidance.
- Reindexing an incompatible index with `Force` recreates all index tables, including dimension-dependent vector storage, writes the current manifest, and indexes all discovered documents.
- `Force` does not recreate a compatible index.
- Embedding vectors whose dimensions differ from the effective configured dimensions are rejected before storage.

## Verification

- Integration-test migrations, FTS5, vector table operations, indexing, search services, and semantic pointer reader.
- Integration-test model and dimension incompatibility, forced recreation, current manifest persistence, and initialization without manifest mutation.
- Unit-test RRF ranking.
