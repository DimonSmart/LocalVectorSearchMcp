# 0003.spec-markdown-model

## Intent

Markdown documents are normalized, parsed into logical elements, addressed with stable semantic pointers, and grouped into chunks suitable for embeddings and search.

- 0004.spec-index-storage-and-search.md

## Behavior

The MVP indexes only `.md` files. Markdown text normalization uses LF line endings, trims trailing whitespace, preserves fenced code block content reasonably, and collapses more than two blank lines.

Supported element kinds are `Document`, `FrontMatter`, `Heading`, `Paragraph`, `CodeBlock`, `ListItem`, `Table`, and `BlockQuote`. The MVP must parse document, front matter, headings, paragraphs, and fenced code blocks. Lists, tables, and blockquotes may initially be represented as paragraphs.

Semantic pointers are stable logical addresses inside a Markdown document. External pointers use `relative/path.md::pointer`; internal pointers use forms such as `frontmatter`, `p1`, `code1`, `1`, `1.p1`, `1.code1`, and `1.2.p3`. Section numbers follow heading order and nesting. Public MCP responses expose relative path plus semantic pointer, never a synthetic document id.

Chunks are built from parsed elements, do not cross document boundaries, keep code blocks intact, respect configured maximum bytes when possible, respect maximum element count, include heading context when configured, and produce stable embedding text hashes.

## Architecture And Patterns

Core defines semantic pointer records, parser and validator APIs, Markdown element DTOs, chunk DTOs, chunker interfaces, and embedding text builder logic. Infrastructure provides the Markdown parser implementation.

`MarkdownSourceDocument` contains relative path, absolute path, normalized Markdown, content hash, and last write time. Markdown domain and response DTOs do not carry a knowledge-base name because a server instance owns only one project source.

## Non-goals

Advanced table/list/blockquote parsing and line-number-only addressing are outside MVP scope.

## Acceptance Criteria

- Pointer validation accepts the specified pointer forms and rejects malformed values.
- Parser detects front matter, headings, nested headings, paragraphs, code blocks, heading paths, and outline entries.
- Pointer generation remains stable when unrelated document content is unchanged.
- Chunks include path, heading context, element text, and stable text hash.
- Markdown documents, elements, chunks, slices, and search documents contain no knowledge-base selector or name.

## Verification

- Unit-test text normalization, pointer parsing, Markdown parsing, chunking, and embedding text generation.
