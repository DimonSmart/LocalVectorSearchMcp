# 0006.adr-mcp-sqlite-hybrid-architecture

## Status

Accepted

## Context

The product must be a local MCP server for Markdown knowledge bases with lexical and semantic retrieval. It needs a local embedded store, official MCP integration, and robust ranking without mixing incompatible score scales.

## Decision

Use the official C# `ModelContextProtocol` SDK for MCP stdio hosting and tool registration. Use SQLite for local storage, SQLite FTS5 for lexical search, sqlite-vec for vector storage and nearest-neighbor search, OpenAI-compatible embeddings for semantic vectors, and Reciprocal Rank Fusion for hybrid ranking.

Core remains free of SQLite, MCP SDK, HTTP client, filesystem, and parser implementation dependencies. Infrastructure owns adapters and persistence. Server owns hosting and tool registration.

## Alternatives Considered

- Manual JSON-RPC/MCP implementation: rejected because protocol transport, discovery, and schema generation are SDK responsibilities.
- Manual vector search in C#: rejected for production because MVP requires sqlite-vec.
- Directly combining BM25, cosine, or vector distance scores: rejected because the score scales are not comparable.
- JSON configuration: rejected for MVP because YAML is the only supported configuration format.
- Separate embedding cache table: rejected for MVP because embeddings are part of the chunk index.

## Consequences

The implementation must load and validate sqlite-vec availability for vector operations, keep FTS and vector row ids aligned with chunk ids, and isolate MCP tools from infrastructure details.

Changing embedding model or dimensions can require forced index rebuild. Hybrid ranking behavior is predictable because it operates on result ranks rather than raw score magnitudes.
